# TerraSoviet Data Rescue — Track 1: NLP & Document AI

Пайплайн для извлечения структурированных данных из отсканированных
советских геологических отчётов. Превращает сканы PDF в машиночитаемый JSON.

## Что делает

1. Берёт PDF-отчёты из папки `input/`
2. Распознаёт кириллический текст (OCR через Tesseract)
3. Парсит **каталоги координат** в структурированные строки (regex по табличной разметке)
4. Извлекает геологические сущности (минералы, участки, координаты, содержания)
5. Сохраняет структурированный JSON в `output/`
6. Оценивает уверенность OCR и помечает слабо распознанные документы флагом
   `needs_review` — честно отделяет надёжные данные от требующих ручной проверки
7. Логирует всё, что не удалось обработать — ничего не теряется молча

> 📂 Готовые примеры «вход → выход» (без запуска) — в папке [`examples/`](examples/):
> настоящий PDF-каталог и два сгенерированных результата — извлечение координат
> (каталог) и извлечение минералов/содержаний (протокол).

> ⚠️ **Про уверенность OCR.** Сканы — машинопись начала 1970-х на выцветшей бумаге,
> поэтому средняя уверенность распознавания по ним объективно низкая (~50%), и
> пайплайн честно помечает такие документы `needs_review`. Это не дефект, а
> работающий индикатор качества: ненадёжное помечается (а сомнительные значения —
> `valid: false` с сохранением сырого текста), но структура всё равно извлекается.
> Подробнее — в [`examples/README.md`](examples/README.md).

## Подход

Решение спроектировано под критерии хакатона:
- **Масштабируемость**: обрабатывает ЛЮБОЕ число PDF в папке `input/` через цикл,
  без хардкода имён файлов. Добавь хоть 10 000 файлов — запустится так же.
- **Воспроизводимость**: пути относительные, зависимости в `requirements.txt`,
  запуск одной командой.
- **Надёжность**: каждый файл в try/except. Один битый скан не роняет весь пайплайн —
  он попадает в лог ошибок, остальные обрабатываются.

Принцип (совет ментора): лучше обработать 80% документов хорошо и честно отметить
остальные как необработанные, чем хардкодить правила под каждый редкий случай.

### Границы решения (что осознанно НЕ делается)

- **Таблицы**: структурируются именно *каталоги координат* (самая частая и
  ценная табличная структура в этих отчётах). Прочие таблицы (например, химические
  составы) сохраняются целиком в поле `raw_text` каждого JSON — их можно достать
  оттуда, но они не разложены по колонкам. Это сознательный выбор в пользу
  масштабируемости, а не недоработка: жёсткие правила под каждый формат таблицы
  плохо переносятся на скрытый датасет.
- **Датум координат**: координаты извлекаются «как есть», в исходном датуме
  документа (как правило Пулково-1942). Перевод в **WGS84** — это Трек 3 и в данном
  решении Трека 1 намеренно не выполняется. То есть `lat_decimal`/`lon_decimal` и
  GeoJSON — это десятичная запись исходных координат, а не перепроецированные WGS84.

## Установка

Нужны две системные зависимости: **Tesseract** (с русским языковым пакетом) и
**poppler** (его использует pdf2image для рендера PDF).

```bash
# Ubuntu/Debian:
sudo apt-get install tesseract-ocr tesseract-ocr-rus poppler-utils

# macOS:
brew install tesseract tesseract-lang poppler
```

**Windows:**
1. Установить Tesseract (сборка UB Mannheim:
   https://github.com/UB-Mannheim/tesseract/wiki), при установке отметить
   русский язык (Russian).
2. Скачать poppler для Windows (https://github.com/oschwartz10612/poppler-windows/releases)
   и добавить его папку `bin` в `PATH`.
3. Если `tesseract.exe` не в `PATH`, указать путь к нему через переменную
   окружения перед запуском:
   ```powershell
   $env:TESSERACT_CMD = "C:\Program Files\Tesseract-OCR\tesseract.exe"
   ```

Затем python-зависимости (для всех ОС):

```bash
pip install -r requirements.txt
```

## Запуск

```bash
# положи PDF-файлы в папку input/, затем:
python main.py

# или укажи свои папки:
python main.py --input путь/к/сканам --output путь/к/результату

# принудительно пересчитать уже обработанные файлы:
python main.py --overwrite
```

## Тестирование

Логика парсинга и извлечения покрыта тестами, которые **не требуют Tesseract/poppler**
(работают на синтетическом тексте) — их можно запустить где угодно:

```bash
pip install pytest
python -m pytest tests/
```

Тесты проверяют, в том числе: нормализацию OCR-подмен цифр в координатах, отсев
шумовых строк, валидацию диапазонов, что номер отчёта не путается с годом, и —
отдельно — что **битый или пустой PDF не роняет прогон**, а попадает в `_errors.json`.

Помимо юнит-тестов, пайплайн прогонялся end-to-end на реальных документах отчёта
№25834 **разных типов**: каталог координат (табличные данные → координаты) и
протокол (сплошной текст → минералы и содержания). Оба примера лежат в
[`examples/`](examples/). Это сознательная проверка на разнородном входе: судьи
запускают код на скрытом похожем датасете, поэтому важно, что пайплайн не заточен
под один формат документа.

## Структура проекта

```
trackone/
├── examples/           # готовый пример вход→выход (PDF + JSON/MD/GeoJSON), можно смотреть без запуска
├── input/              # сюда кладутся PDF-сканы (создай и положи файлы)
├── output/             # сюда пишется JSON/Markdown-результат + лог ошибок (создаётся автоматически)
├── main.py             # точка входа, обходит все файлы
├── ocr.py              # PDF → текст (Tesseract, предобработка, авто-поворот)
├── parse_catalog.py    # парсинг таблиц-каталогов координат
├── entities.py         # извлечение геологических сущностей
├── requirements.txt
└── README.md
```

## Формат выхода

Для каждого PDF создаётся `<имя>.json`:

```json
{
  "source_file": "каталог_координат_25834.pdf",
  "report_id": "25834",
  "processed_at": "2026-06-27T12:00:00",
  "ocr": {
    "pages": 4,
    "failed_pages": 0,
    "mean_confidence": 78.5,
    "needs_review": false
  },
  "coordinates": [
    {
      "id": "1",
      "name": "Ю.Жуманай",
      "lat_dms": "48°03'",
      "lon_dms": "71°16'",
      "lat_decimal": 48.05,
      "lon_decimal": 71.266667
    }
  ],
  "entities": {
    "minerals": ["висмут", "золото", "серебро"],
    "grades_g_per_t": ["23,9", "13,2"],
    "coordinates_in_text": [
      {"value": "48°03'", "type": "с.ш.", "valid": true, "decimal": 48.05},
      {"value": "71°16'", "type": "в.д.", "valid": true, "decimal": 71.266667}
    ]
  },
  "raw_text": "...полный распознанный текст..."
}
```

Дополнительно создаются:
- `<имя>.md` — человекочитаемое саммари документа (Markdown);
- `output/all_coordinates.geojson` — все координаты из всех документов одним
  файлом (открывается на [geojson.io](https://geojson.io));
- `output/summary.json` и `output/summary.md` — **сводка по всему корпусу**:
  сколько документов обработано, сколько помечено `needs_review`, какие минералы
  встречаются и в скольких документах, сколько валидных координат собрано всего.
  Даёт обзор спасённого архива одним взглядом;
- `output/_errors.json` — лог файлов, которые не удалось обработать.
