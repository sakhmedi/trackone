# TerraSoviet Data Rescue · Трек 1: NLP и Document AI

Автоматизированный конвейер, который превращает отсканированные советские
геологические отчёты (PDF) в структурированные, машиночитаемые данные: JSON,
Markdown и GeoJSON. Проект распознаёт кириллический текст, разбирает табличные
каталоги координат, извлекает геологические сущности (минералы, участки,
содержания металлов) и честно отмечает всё, что распозналось ненадёжно.

## Что делает проект

Конвейер выполняет полный путь от скана до структурированных данных:

1. Берёт все PDF-файлы из входной папки (рекурсивно, с любым числом файлов).
2. Распознаёт кириллический текст через Tesseract OCR с предобработкой скана.
3. Разбирает каталоги координат в структурированные строки вида
   «участок, широта, долгота».
4. Извлекает геологические сущности: минералы и металлы, содержания в г/т,
   координаты, упомянутые в тексте, номер отчёта.
5. Оценивает среднюю уверенность OCR и помечает слабо распознанные документы
   флагом `needs_review`.
6. Сохраняет результат по каждому документу в JSON и Markdown.
7. Собирает сводные артефакты по всему набору: карту координат (GeoJSON),
   сводку по корпусу и список уникальных участков после дедупликации.
8. Логирует каждый файл, который не удалось обработать, не прерывая прогон.

> **Важно про уверенность OCR.** Сканы представляют собой машинопись начала
> 1970-х годов на выцветшей бумаге, поэтому средняя уверенность распознавания по
> ним объективно низкая (около 50 процентов). Конвейер не выдаёт такие данные за
> точные: он помечает документ как `needs_review`, а сомнительные значения как
> `"valid": false` с сохранением сырого текста. При этом структура всё равно
> извлекается. Низкая уверенность здесь работает как индикатор качества, а не как
> сбой распознавания.

## Как это работает

Конвейер разбит на независимые модули, каждый отвечает за один этап.

### Этап 1. Распознавание (`ocr.py`)

Каждая страница PDF рендерится в изображение (300 dpi) и проходит предобработку
под машинопись 1970-х:

* автоматический поворот страницы по данным Tesseract OSD (часть приложений
  отсканирована повёрнутой на 90 градусов);
* перевод в оттенки серого и выравнивание контраста (это лечит неравномерное
  выцветание).

Бинаризацию изображения мы намеренно оставляем самому Tesseract: на реальных
сканах этого отчёта собственная бинаризация движка по серому изображению даёт
заметно больше распознанных строк, чем ручной порог. Большие документы
рендерятся батчами, чтобы не держать сотни страниц в памяти. Каждая страница
обрабатывается в собственном блоке обработки ошибок: одна битая страница не
роняет весь документ.

### Этап 2. Разбор каталога координат (`parse_catalog.py`)

Табличные строки каталога разбираются регулярными выражениями. Для таблиц это
точнее, быстрее и полностью воспроизводимо. Решение устойчиво к типичным ошибкам
кириллического OCR: цифры в координатах часто распознаются как похожие буквы
(0 и О, 1 и l, 3 и З, 8 и В). Такие символы допускаются в позициях координат и
нормализуются обратно в цифры. Каждая строка проверяется на допустимый диапазон
градусов и минут. Нечитаемые или неправдоподобные строки не выбрасываются молча,
а помечаются как `"valid": false`.

### Этап 3. Извлечение сущностей (`entities.py`)

Минералы и металлы ищутся по основе слова, поэтому распознаются во всех падежах
(«золота», «серебром», «медный»). Содержания металлов извлекаются по шаблону
вида «13,2 г/т» с отсевом значений, похожих на календарный год. Номер отчёта
определяется по контексту в тексте и по имени файла, не путая номер с годом.

### Этап 4. Сборка и сводки (`main.py`)

Точка входа обходит все файлы, пишет результат по каждому документу и в конце
собирает сводные артефакты по всему корпусу. Прогон поддерживает докатку: если
для файла результат уже есть, он пропускается, поэтому прогон на тысячах файлов
можно перезапустить после сбоя.

## Требования

* **Python 3.9 или новее.**
* **Tesseract OCR** с установленным русским языковым пакетом (`rus`).
* **Poppler** (его использует библиотека `pdf2image` для рендера PDF).
* Python-зависимости из `requirements.txt` (`pytesseract`, `pdf2image`,
  `Pillow`).

## Установка

Сначала установите две системные зависимости (Tesseract и Poppler), затем
Python-пакеты. Ниже инструкция для каждой операционной системы.

### Windows

1. Установите Tesseract из сборки UB Mannheim:
   [github.com/UB-Mannheim/tesseract/wiki](https://github.com/UB-Mannheim/tesseract/wiki).
   В процессе установки обязательно отметьте русский язык (Russian).
2. Скачайте Poppler для Windows:
   [github.com/oschwartz10612/poppler-windows/releases](https://github.com/oschwartz10612/poppler-windows/releases)
   и добавьте папку `bin` из архива в переменную окружения `PATH`.
3. Если `tesseract.exe` не виден в `PATH`, укажите путь к нему через переменную
   окружения перед запуском (PowerShell):

   ```powershell
   $env:TESSERACT_CMD = "C:\Program Files\Tesseract-OCR\tesseract.exe"
   ```

### macOS

Установите обе зависимости через Homebrew:

```bash
brew install tesseract tesseract-lang poppler
```

Пакет `tesseract-lang` содержит русский язык. Tesseract и Poppler после
установки доступны в `PATH`, дополнительная настройка не нужна.

### Linux (Ubuntu или Debian)

Установите пакеты через apt:

```bash
sudo apt-get update
sudo apt-get install tesseract-ocr tesseract-ocr-rus poppler-utils
```

Пакет `tesseract-ocr-rus` добавляет русский язык, `poppler-utils` содержит
рендер PDF.

### Python-зависимости (для всех систем)

После установки системных зависимостей установите Python-пакеты:

```bash
pip install -r requirements.txt
```

## Запуск

Пошаговый сценарий полного прогона:

1. Создайте папку `input` в корне проекта и положите в неё PDF-файлы отчётов.
2. Запустите конвейер:

   ```bash
   python main.py
   ```

3. Результаты появятся в папке `output` (она создаётся автоматически): по файлу
   `JSON` и `Markdown` на каждый документ плюс сводные артефакты.

Вы можете указать собственные папки вместо стандартных `input` и `output`. Это
основной сценарий для проверяющего, у которого данные лежат в другом месте:

```bash
python main.py --input путь/к/сканам --output путь/к/результату
```

По умолчанию уже обработанные файлы пропускаются (докатка). Чтобы пересчитать
их заново, добавьте флаг `--overwrite`:

```bash
python main.py --overwrite
```

## Аргументы командной строки

| Аргумент | По умолчанию | Назначение |
| :--- | :--- | :--- |
| `--input` | `input` | Папка с входными PDF-файлами (обходится рекурсивно). |
| `--output` | `output` | Папка для результатов (создаётся автоматически). |
| `--overwrite` | выключен | Пересчитать файлы, для которых результат уже есть. |

Пути задаются только аргументами, в коде нет жёстко прописанных путей. Это
позволяет запускать конвейер на любом датасете без правки исходников.

## Что получается на выходе

### Результат по каждому документу

Для каждого PDF создаётся файл `<имя>.json` со следующей структурой:

```json
{
  "source_file": "каталог_координат_25834.pdf",
  "report_id": "25834",
  "processed_at": "2026-06-27T12:00:00",
  "ocr": {
    "pages": 4,
    "failed_pages": 0,
    "mean_confidence": 78.5,
    "needs_review": false
  },
  "coordinates": [
    {
      "id": "1",
      "name": "Ю.Жуманай",
      "lat_dms": "48°03'",
      "lon_dms": "71°16'",
      "lat_decimal": 48.05,
      "lon_decimal": 71.266667
    }
  ],
  "entities": {
    "minerals": ["висмут", "золото", "серебро"],
    "grades_g_per_t": ["23,9", "13,2"],
    "coordinates_in_text": [
      {"value": "48°03'", "type": "с.ш.", "valid": true, "decimal": 48.05},
      {"value": "71°16'", "type": "в.д.", "valid": true, "decimal": 71.266667}
    ]
  },
  "raw_text": "...полный распознанный текст..."
}
```

Рядом создаётся файл `<имя>.md` с человекочитаемым саммари того же документа.

### Сводные артефакты по всему набору

Помимо результата по каждому документу, в конце прогона собираются файлы по всему
корпусу:

| Файл | Содержимое |
| :--- | :--- |
| `all_coordinates.geojson` | Все валидные координаты из всех документов в одном файле. Открывается на [geojson.io](https://geojson.io). |
| `summary.json` и `summary.md` | Сводка по корпусу: сколько документов обработано, сколько помечено `needs_review`, какие минералы встречаются и в скольких документах, сколько валидных координат собрано всего. |
| `sites.json` и `sites.md` | Уникальные участки после дедупликации между документами. Один и тот же участок (совпадает название и координаты), встретившийся в нескольких отчётах, схлопывается в одну запись со списком источников. Отдельно выносятся конфликты: одно название с разными координатами, как сигнал для ручной проверки. |
| `_errors.json` | Лог файлов, которые не удалось обработать, с текстом ошибки. Создаётся только при наличии сбоев. |

## Тестирование

Логика разбора и извлечения покрыта тестами на `pytest`. Тесты работают на
синтетическом тексте и не требуют установленного Tesseract или Poppler, поэтому
запускаются в любом окружении:

```bash
pip install pytest
python -m pytest tests/
```

Тесты покрывают, в частности: нормализацию OCR-подмен цифр в координатах, отсев
шумовых строк, валидацию диапазонов, корректное определение номера отчёта (без
путаницы с годом), дедупликацию участков. Отдельный тест проверяет, что битый или
пустой PDF не роняет прогон, а попадает в `_errors.json`.

Дополнительно конвейер прогонялся на реальных документах отчёта номер 25834 двух
разных типов: каталог координат (табличные данные) и протокол (сплошной текст).
Оба результата лежат в папке [`examples/`](examples/). Это проверка на
разнородном входе, так как проверяющие запускают код на скрытом похожем датасете.

## Инженерный подход

Решение спроектировано под критерии хакатона.

* **Масштабируемость.** Конвейер обрабатывает любое число PDF в папке через цикл,
  без жёстко прописанных имён файлов. Добавьте хоть 10 000 файлов, и прогон
  пройдёт так же.
* **Воспроизводимость.** Пути относительные и задаются аргументами, зависимости
  перечислены в `requirements.txt`, запуск выполняется одной командой.
* **Надёжность.** Каждый файл обрабатывается в отдельном блоке защиты от ошибок.
  Один битый скан не роняет весь конвейер: он попадает в лог ошибок, а остальные
  файлы обрабатываются. Прогон поддерживает докатку после сбоя.

В основе лежит совет ментора: лучше обработать большую часть документов хорошо и
честно отметить остальные как требующие проверки, чем прописывать жёсткие правила
под каждый редкий случай.

## Границы решения

Несколько вещей не делаются осознанно, чтобы решение хорошо переносилось на
скрытый датасет.

* **Таблицы.** Структурируются именно каталоги координат как самая частая и
  ценная табличная структура в этих отчётах. Прочие таблицы (например, химические
  составы) сохраняются целиком в поле `raw_text` каждого JSON. Их можно достать
  оттуда, но они не разложены по колонкам. Это выбор в пользу масштабируемости:
  жёсткие правила под каждый формат таблицы плохо переносятся на новые данные.
* **Датум координат.** Координаты извлекаются в исходном датуме документа (как
  правило, Пулково-1942). Перевод в WGS84 относится к Треку 3 и в данном решении
  Трека 1 намеренно не выполняется. Поля `lat_decimal` и `lon_decimal`, а также
  GeoJSON, содержат десятичную запись исходных координат, а не перепроецированные
  значения WGS84.
