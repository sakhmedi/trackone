# Pipeline example

This folder contains a ready-made "input and output" example, so you can see the
result without running anything. Everything here was generated by a real run of
`main.py` on an actual report, not written by hand.

## What is in the folder

Two examples from a single report, number 25834. They show different halves of
the pipeline.

| File | What it is |
| :--- | :--- |
| `каталог_координат_25834.pdf` | Input: a scanned Soviet coordinate catalog (4 pages). |
| `каталог_координат_25834.json` and `.md` | Output: extraction of site coordinates (the geospatial part). |
| `протокол_25834.json` and `.md` | Output: entity extraction (minerals, grades in g/t) from 12 pages of continuous protocol text. |
| `all_coordinates.geojson` | Output: all valid coordinates in a single file (opens on [geojson.io](https://geojson.io)). |
| `summary.json` and `summary.md` | Output: corpus summary (2 documents, both `needs_review`, 5 valid coordinates, minerals across the whole set). |
| `sites.json` and `sites.md` | Output: sites after cross-document deduplication and name conflicts. There are 5 unique sites here. Cross-document duplicates will appear when a single site occurs in several reports. |

> The `протокол_25834.pdf` file itself (8 MB) is not committed to the repository,
> to keep it from bloating. The JSON and Markdown here were produced by running
> `main.py` on it.

## How to reproduce this

From the repository root, run:

```bash
python main.py --input examples --output examples
```

An already-processed file will be skipped. To recompute it from scratch, add the
`--overwrite` flag.

## Why both examples are flagged `needs_review`

These are honest examples, not polished ones. OCR recognized both documents with
a mean confidence of about 50 percent (catalog 49.7, protocol 50.8), so both are
automatically flagged `"needs_review": true`.

This is exactly as it should be: it is an objective property of the dataset, not
a defect of the pipeline. The scans are early-1970s typewritten pages on faded
paper. Even a human reads them with difficulty, and no OCR will give a confident
90 percent on them. The value of the solution lies precisely in what it does
next.

* **Does not pass unreliable data off as reliable.** The pipeline honestly sets
  the flag and the mean confidence, so an operator knows exactly what to check.
  This follows the mentor's advice: better to honestly flag the unreliable than
  to hard-code rules for every case.
* **Still extracts structure** from such input: site coordinates from the
  catalog, and minerals (gold, silver, bismuth, copper, zinc, quartz) and grades
  in g/t from the protocol. Real OCR errors are visible in the output too, but
  they are flagged (`valid: false`, raw values preserved), not silently dropped.

In other words, low confidence here works as a quality indicator, not as a
recognition failure.
