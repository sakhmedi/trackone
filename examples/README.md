# Пример работы конвейера

Эта папка содержит готовый пример «вход и выход», чтобы можно было увидеть
результат, ничего не запуская. Всё здесь сгенерировано реальным прогоном
`main.py` на настоящем отчёте, а не написано вручную.

## Что лежит в папке

Два примера из одного отчёта номер 25834. Они показывают разные половины
конвейера.

| Файл | Что это |
| :--- | :--- |
| `каталог_координат_25834.pdf` | Вход: отсканированный советский каталог координат (4 страницы). |
| `каталог_координат_25834.json` и `.md` | Выход: извлечение координат участков (геопространственная часть). |
| `протокол_25834.json` и `.md` | Выход: извлечение сущностей (минералы, содержания в г/т) из 12 страниц сплошного текста протокола. |
| `all_coordinates.geojson` | Выход: все валидные координаты одним файлом (открывается на [geojson.io](https://geojson.io)). |
| `summary.json` и `summary.md` | Выход: сводка по корпусу (2 документа, оба `needs_review`, 5 валидных координат, минералы по всему набору). |
| `sites.json` и `sites.md` | Выход: участки после дедупликации между документами и конфликты названий. Здесь 5 уникальных участков. Дубли между документами появятся, когда один участок встретится в нескольких отчётах. |

> Сам файл `протокол_25834.pdf` (8 МБ) в репозиторий не кладётся, чтобы не
> раздувать его. Лежащие здесь JSON и Markdown получены прогоном `main.py` на нём.

## Как это воспроизвести

Из корня репозитория выполните:

```bash
python main.py --input examples --output examples
```

Уже обработанный файл будет пропущен. Чтобы пересчитать его заново, добавьте флаг
`--overwrite`.

## Почему оба примера помечены `needs_review`

Это честные примеры, а не причёсанные. Оба документа OCR распознал со средней
уверенностью около 50 процентов (каталог 49.7, протокол 50.8), поэтому оба
автоматически помечены `"needs_review": true`.

Так и должно быть: это объективное свойство датасета, а не дефект конвейера.
Сканы представляют собой машинопись начала 1970-х годов на выцветшей бумаге.
Даже человек читает их с трудом, и ни один OCR не даст по ним уверенных 90
процентов. Ценность решения как раз в том, что оно делает следующее.

* **Не выдаёт ненадёжное за надёжное.** Конвейер честно ставит флаг и среднюю
  уверенность, чтобы оператор знал, что именно нужно проверить. Это следует
  совету ментора: лучше честно отметить ненадёжное, чем прописывать жёсткие
  правила под каждый случай.
* **Всё равно извлекает структуру** из такого входа: из каталога координаты
  участков, из протокола минералы (золото, серебро, висмут, медь, цинк, кварц) и
  содержания в г/т. В выходе видны и реальные ошибки OCR, но они помечены
  (`valid: false`, сырые значения сохранены), а не выброшены молча.

То есть низкая уверенность здесь работает как индикатор качества, а не как провал
распознавания.
